Overview

项目概览

Superpowers 是一个结构化的、强制纪律执行的工作流系统，引导 AI 编码代理完成完整的开发周期

⚡

是什么

一套以 "Skills" 为核心的可组合插件系统，为 AI 代理提供从头脑风暴、设计、计划、实现、代码审查到分支管理的完整开发工作流。Skills 基于上下文自动触发，无需用户手动调用。

🎯

解决什么问题

AI 代理在自由发挥时容易跳过测试、忽略根因分析、声称修复但未验证。Superpowers 通过 "铁律"（Iron Laws）和 "红旗检测"（Red Flags）强制代理遵循 TDD、系统性调试、完成前验证等工程规范。

🛠

核心特征

零依赖设计 / 测试驱动开发为铁律 / 系统性优于即兴 / 复杂度最小化 / 证据优于声称 / 多平台插件架构 / 子代理驱动的并行开发

Philosophy

设计理念

每个设计决策背后都有经过实战检验的理由

🔮

零外部依赖

核心插件不引入任何第三方依赖，保持极致的可移植性和稳定性

🔴

TDD 铁律

"没有失败测试就不写生产代码" —— 这不是建议，是强制执行的规则

🔍

证据优于声称

所有完成声明必须附带新鲜的验证证据，不接受 "应该可以" 或缓存结果

🧰

系统性优于即兴

调试遵循根因分析，不允许随机尝试修复；开发遵循计划，不允许跳步骤

⚖

YAGNI 原则

只做当前需要的最小改动，不过度工程化，三行重复代码好过提前抽象

🤝

"你的人类伙伴"

刻意使用 "your human partner" 而非 "the user" —— 这是精心设计的行为塑造语言

Workflow

端到端工作流

Skills 基于上下文自动触发，引导代理完成完整开发周期

1

会话初始化

session-start 钩子注入 using-superpowers 技能上下文

2

头脑风暴

澄清需求 → 探索方案 → 编写设计文档 → 用户审批

3

编写计划

将设计拆分为小步骤的 TDD 任务，每步含具体代码和命令

4

实现开发

子代理并行执行 + TDD 强制 + 系统性调试 + 双阶段审查

5

验证 & 审查

完成前验证 + 代码审查 + 规格合规检查

6

分支收尾

测试通过 → 合并/PR/保留/丢弃四选一 → 清理 worktree

Core Skills

14 项核心技能

每项技能都是塑造代理行为的代码，而非简单的文档

💡

Brainstorming 头脑风暴

在任何创造性工作之前激活。通过逐步澄清问题、探索 2-3 种方案、编写设计文档，确保在动手之前充分理解需求。

触发：功能设计、组件创建、功能修改之前
📝

Writing Plans 编写计划

将设计规格转化为可执行的实现计划。每个任务控制在 2-5 分钟，包含精确文件路径、完整代码块和具体命令，杜绝 TBD/TODO 等占位符。

触发：有了规格/需求，准备拆解实现步骤时
🤖

Subagent-Driven Dev 子代理驱动开发

为每个任务分派独立的实现子代理，完成后进行双阶段审查（规格合规 + 代码质量）。支持智能模型选择：简单任务用廉价模型，架构决策用最强模型。

触发：执行含独立任务的实现计划时
▶

Executing Plans 执行计划

加载实现计划并逐步执行，适用于不支持子代理的平台。在每个检查点验证进度，完成后触发分支收尾流程。

触发：在独立会话中执行已编写的实现计划时
🔴

Test-Driven Development 测试驱动开发

强制执行 RED → GREEN → REFACTOR 循环。包含大量 "合理化思维" 检测表和红旗列表，封堵所有常见的跳过测试借口。

触发：在编写任何生产代码之前
铁律：没有失败测试就不写生产代码
🐛

Systematic Debugging 系统性调试

四阶段方法论：根因调查 → 模式分析 → 假设检验 → 实施修复。3 次失败修复后必须停下来质疑架构。附带多个实用脚本和参考文档。

触发：任何 bug、测试失败或意外行为
铁律：不做根因分析就不修复
✅

Verification Before Completion 完成前验证

在声称任何工作已完成之前，必须运行验证命令、读取完整输出、确认结果。不接受基于记忆或缓存的 "应该没问题"。

触发：声称工作完成、修复成功、测试通过之前
铁律：没有新鲜验证证据就不声称完成
🔍

Requesting Code Review 请求代码审查

获取 Git SHA，使用模板分派代码审查子代理。审查员会对照计划/规格检查实现，发现问题分为关键/重要/建议三级。

触发：完成任务、实现功能、合并前
💬

Receiving Code Review 接收代码审查

收到反馈时的行为准则：禁止表演性附和（"说得太对了！"），验证后再实施，审查员错误时用技术论据反驳，对 "专业" 建议执行 YAGNI 检查。

触发：收到反馈，尤其是不确定或有疑问时
⚖

Dispatching Parallel Agents 并行代理调度

识别无共享状态的独立任务，创建聚焦的代理任务描述，并行分派执行，最后审查和集成结果。

触发：2+ 个无共享状态的独立任务
🌳

Using Git Worktrees 使用 Git 工作树

为功能开发创建隔离的工作树，检查现有目录规范、确保 gitignore、运行项目初始化、验证测试基线。

触发：开始需要隔离的功能开发、执行计划前
🏁

Finishing Dev Branch 开发分支收尾

验证测试全部通过后，精确提供四个选项：本地合并、创建 PR、保持原样、丢弃。执行选择后清理工作树。

触发：实现完成且所有测试通过
🚀

Using Superpowers 技能调度引导

每次会话启动时注入。核心规则：即使只有 1% 的适用概率也要调用相关技能。包含红旗检测表、技能优先级、平台适配说明。

触发：每次会话开始（通过 hook 自动注入）
✍

Writing Skills 技能编写

用 TDD 方法论编写技能本身：基线测试（无技能）→ 编写最小技能 → 堵漏洞。包含 Claude 搜索优化（CSO）、说服力原则、测试方法论。

触发：创建、编辑或验证技能时
Hooks

Hooks 系统

会话启动时自动注入技能上下文，跨平台兼容

⚡

Session Start Hook

核心钩子。读取 using-superpowers/SKILL.md，转义为 JSON 格式，根据平台环境变量自动选择输出格式（Claude Code / Cursor / SDK 标准）。

事件：startup / clear / compact
🖥

跨平台兼容

run-hook.cmd 是一个多语言文件（同时是 batch 和 bash），在 Windows 上自动查找 Git Bash 执行钩子脚本，在 Unix 上直接运行。

搜索路径：Program Files → x86 → PATH
🔄

平台检测逻辑

通过环境变量自动识别平台：CURSOR_PLUGIN_ROOT → Cursor 格式 / CLAUDE_PLUGIN_ROOT → Claude Code 格式 / 其他 → SDK 标准格式。同时检测旧版配置并提示迁移。

Multi-Platform

多平台支持

平台 插件机制 配置文件 特殊适配
