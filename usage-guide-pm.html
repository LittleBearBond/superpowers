用 Superpowers
让 AI 像专业工程师一样工作

不需要写代码，不需要懂技术细节。只需要描述你想要什么，Superpowers 会让 AI 按照专业软件工程流程帮你实现。

📚  本指南假设你使用 Claude Code，全程无需编码经验
What is it

Superpowers 是什么？

一句话：它是 AI 编程助手的 "工作纪律系统"。

想象一下这个场景

你对 Claude 说："帮我做一个用户反馈收集功能。"

没有 Superpowers 时，Claude 可能直接开始写代码 — 它猜测你想要什么、跳过测试、写完后说 "搞定了"。你不知道它做了什么，出了问题也说不清。

有了 Superpowers 后，Claude 会像一个专业工程师那样工作：先问你一系列问题搞清楚需求，提出几个方案让你选，写出设计文档给你确认，然后才开始一步步实现 — 而且每一步都有测试和审查。

类比：如果说 Claude 是一个新入职的工程师，那 Superpowers 就是他的 "入职培训手册" + "工作流程规范" + "代码审查制度"。它不会改变 Claude 的能力，但会改变 Claude 的工作方式。
Why it matters

对产品经理的价值

你不需要关心背后的技术原理，只需要知道它能帮你做什么。

💬

需求不会被误解

Claude 会逐个问题澄清你的想法，而不是猜测你想要什么。你会看到一份设计文档，确认无误后才开始开发。

📄

过程可见、可追溯

每个阶段都有文档产出：设计文档、实现计划、代码审查报告。你可以随时查看进度，知道做了什么、为什么这样做。

✅

质量有保障

自动执行测试驱动开发、代码审查、完成前验证。Claude 不能说 "应该没问题" — 必须拿出证据证明代码可用。

🕑

可以自主运行数小时

一旦你批准了设计和计划，Claude 可以自主执行几个小时而不偏离目标。你不需要一直盯着。

Getting Started

3 分钟完成安装

只需要一条命令，不需要任何编程知识。

步骤一：打开 Claude Code

在终端中启动 Claude Code。如果你已经在使用 Claude Code，直接进行下一步。

步骤二：输入安装命令

在 Claude Code 中输入以下内容并回车：

/plugin install superpowers@claude-plugins-official

等待安装完成，看到成功提示即可。

步骤三：开始新会话

安装完成后，必须开始一个新会话（关闭当前会话再打开，或输入 /clear）。Superpowers 在新会话启动时自动加载。

验证安装成功：在新会话中说 "帮我设计一个新功能"。如果 Claude 回复中提到了 "brainstorming" 或开始问你澄清问题（而不是直接写代码），说明安装成功了。
Your Journey

你和 Claude 的完整协作旅程

从 "我想做一个 X" 到交付完成，每一步你会看到什么、需要做什么。

1

头脑风暴阶段

约 10-30 分钟

发生了什么：Claude 不会直接写代码。它会先退一步，像一个产品设计师和你对话。

1

Claude 了解项目现状

它会自动查看你项目中的文件和最近的改动，了解背景。

Claude 在做：查看项目文件、文档、git 历史
2

逐个问你问题

每次只问一个问题，很多时候会给你选项让你选。不会一次扔给你 10 个问题。

你需要做：回答 Claude 的问题，选择你偏好的选项
3

提出 2-3 种方案

Claude 始终会提供多个方案，每个附带优缺点分析和推荐理由。它不会只给你一个方案。

你需要做：选择一个方案，或者说 "我更倾向 A 但是想把 B 的某个优点也加进来"
4

分段展示设计

不会一次性丢一个 50 页的文档。Claude 会分段展示设计：先是整体架构，然后是各个组件，再到数据流、错误处理等。每段结束后它会问你 "这部分看起来对吗？"

你需要做：确认每个部分，提出疑问或修改意见
5

写出设计文档并保存

所有部分确认后，Claude 会把完整设计保存为一个 Markdown 文件。

你需要做：最终审阅设计文档。这是最后的把关 — 确认后 Claude 才会进入实现
关键规则：在你审批设计之前，Claude 不会写任何代码。即使你的需求 "很简单"，也会走这个流程。这是为了确保 Claude 真正理解了你想要什么。
2

计划阶段

约 5-15 分钟

发生了什么：Claude 把设计拆分成一个个小任务，像拼积木一样组织好。每个任务小到可以在几分钟内完成。

1

Claude 创建隔离工作区

在一个独立的分支上工作，不影响已有代码。就像在草稿纸上画，画完再决定要不要正式采用。

Claude 在做：创建 git 工作树、安装项目依赖、验证测试基线
2

写出详细的实现计划

每个任务都包含：要创建/修改哪些文件、具体的代码内容、验证方法。计划文件也会保存下来。

Claude 在做：把设计拆分为 2-5 分钟的小任务，每个任务有明确的验收标准
3

选择执行方式

Claude 会问你怎么执行这个计划。推荐的方式是 "子代理驱动" — Claude 会自动分派小助手来并行完成任务。

你需要做：选择执行方式（一般选推荐的就好）
3

实现阶段

可自主运行数小时

发生了什么：这是 Claude 自主工作的阶段。它会自动完成所有编码、测试、审查工作。你可以去做别的事。

1

逐任务开发

对每个任务，Claude 分派一个 "子代理"（可以理解为一个专注的小助手）来完成。子代理会：

  • 先写测试（证明功能应该怎么工作）
  • 再写代码（让测试通过）
  • 自己检查一遍
Claude 在做：分派子代理 → 实现 → 测试 → 自审 → 提交代码
2

双阶段质量审查

每个任务完成后，Claude 会派两个独立的审查员检查：

  • 审查员 1：检查 "做的对不对" — 实现是否符合计划要求？
  • 审查员 2：检查 "做的好不好" — 代码质量、架构、测试是否到位？
Claude 在做：分派审查子代理，如发现问题则让实现者修复，直到两个审查都通过
3

遇到问题时自动调试

如果某个测试失败或出了 Bug，Claude 不会随便改改碰运气。它有一套严格的四步诊断流程：先找原因，再验证假设，最后才修复。如果修了 3 次还没修好，它会停下来重新审视整个方案。

Claude 在做：根因分析 → 模式对比 → 假设验证 → 定向修复
你可以放手：这个阶段通常不需要你参与。Claude 会自己跑完所有任务。你可以去开会、处理其他工作，回来后检查结果即可。
4

收尾阶段

约 2-5 分钟

发生了什么：所有任务完成且测试通过后，Claude 让你决定怎么处理这些代码。

1

Claude 确认所有测试通过

它会运行完整的测试套件，确认一切正常。如果有失败的测试，会先修复。

2

给你四个选项

你需要选一个：
1. 合并 — 把代码合并到主分支（立即生效）
2. 创建 PR — 推送代码并创建 Pull Request（让团队审查）
3. 先放着 — 保持现状，你稍后再处理
4. 丢弃 — 不要这些代码了（需要输入 "discard" 确认）
推荐做法：如果你在一个团队中工作，选 "创建 PR" 最安全 — 团队成员可以在合并前审查代码。如果是个人项目，"合并" 更快捷。
Examples

对话示例：你和 Claude 怎么交流

真实场景下的对话流程，帮你建立预期。

场景：开发一个用户反馈功能

你
帮我做一个用户反馈收集功能，让用户可以提交问题反馈和功能建议。
Claude
我正在使用 brainstorming 技能来设计这个功能。

我先看一下项目现状...

关于这个反馈功能，我想先了解一个问题：

反馈的主要受众是什么角色？

A) 终端用户（产品的使用者）
B) 内部团队（同事、测试人员）
C) 两者都有
你
A，主要是终端用户
Claude
明白。那用户提交反馈后，你希望怎么处理这些反馈？

A) 只是收集存储，后台看一个列表就行
B) 需要一个管理后台，能标记状态（待处理/处理中/已解决）
C) 需要更完整的流程，包括回复用户、优先级排序等
你
B，我需要能标记状态的管理后台
Claude
好的，我有几个方案：

方案 A（推荐）：内嵌反馈面板
优点：用户无需离开当前页面，体验流畅
缺点：开发量稍大

方案 B：独立反馈页面
优点：开发简单，好维护
缺点：用户需要跳转

方案 C：浮动按钮 + 弹窗
优点：始终可见，随时提交
缺点：会占据界面空间

我推荐方案 A，因为...
你
方案 A 听起来不错，就用这个
Claude
好的，让我来展示设计。先从整体架构开始：

架构概览
前端：一个可滑出的侧边面板组件...
后端：一个 RESTful API...
数据库：一张 feedbacks 表...

这部分设计看起来合理吗？
注意这个模式：Claude 会一个问题一个问题地问你、给你选项让你选、展示方案让你挑、分段展示设计让你确认。你不需要一次性把所有需求写清楚 — Claude 会引导你一步步完善。

Decisions

你需要做哪些决策？

在整个过程中，Claude 会在几个关键节点需要你的判断。

时机Claude 会问你什么怎么回答
需求澄清 关于功能目的、受众、约束的具体问题 选择给出的选项，或用你自己的话描述
方案选择 "推荐方案 A / B / C，你选哪个？" 选一个，或者混搭（"A 的整体方案 + B 的某个特点"）
设计确认 "这部分设计看起来对吗？" "没问题" / "这里我有个疑问..." / "这里我想改一下..."
设计文档审批 "设计文档写好了，请审阅" 看一遍确认没问题就说 "可以了，开始做吧"
执行方式 "用子代理驱动还是内联执行？" 选推荐选项（子代理驱动）即可
收尾操作 "合并 / 创建 PR / 先放着 / 丢弃？" 根据你的工作流选择。团队项目建议选 "创建 PR"
一个原则：如果你不确定某个选项的含义，直接问 Claude。它会用通俗的语言解释。你永远不需要 "假装懂技术"。
Deliverables

每个阶段的产出物

这些是你可以查看、存档、分享给团队的实体文件。

📄 设计文档

产生时机：头脑风暴阶段结束时

保存位置：docs/superpowers/specs/ 目录下

包含内容：功能概述、架构设计、组件定义、数据流、错误处理策略、测试策略

用途：需求确认的凭证、团队沟通的基础、后续维护的参考

📝 实现计划

产生时机：计划阶段结束时

保存位置：docs/superpowers/plans/ 目录下

包含内容：拆解后的任务列表，每个任务有明确的文件路径、代码内容、验证方法

用途：进度跟踪、工作量评估、质量验收的参照

✅ 代码 + 测试

产生时机：实现阶段逐步产出

保存位置：项目代码库中（在独立分支上）

包含内容：功能代码 + 每个功能的自动化测试

用途：每个功能都有测试覆盖，确保后续修改不会无意中破坏已有功能

🔍 审查报告

产生时机：每个任务完成后、全部完成后

包含内容：代码是否符合设计、质量评估、发现的问题及修复情况

用途：质量保障的证据、问题的可追溯性

Comparison

有 vs 没有 Superpowers

同样是让 AI 帮你做功能，体验完全不同。

❌ 没有 Superpowers

  • Claude 猜测你的需求
  • 直接开始写代码
  • 不写测试或很少测试
  • 出 Bug 时随便改改碰运气
  • 说 "搞定了" 但可能没测过
  • 没有文档留下来
  • 一个月后没人记得为什么这样做

✅ 有 Superpowers

  • Claude 逐步澄清需求
  • 先写设计文档你确认
  • 每个功能都有自动化测试
  • 出 Bug 时做根因分析
  • 必须跑通测试才能说 "完成"
  • 设计文档 + 计划 + 审查报告
  • 一个月后打开文档就知道来龙去脉
Best Practices

使用技巧

让 Superpowers 发挥最大价值的实用建议。

🎯 描述 "为什么" 比描述 "怎么做" 更重要

与其说 "用 React 做一个表单"，不如说 "用户需要一个提交反馈的方式，要简单快捷，不打断他们的使用流程"。Claude 会帮你找到最好的实现方式。

⏬ 需求在设计阶段提出，实现阶段别加

在头脑风暴阶段尽量把想法都说出来。一旦计划写好并开始实现，中途加需求会导致返工。如果确实想加，可以说 "我们先完成当前的，然后再做一轮迭代"。

👉 不确定就选推荐选项

Claude 提供选择时通常会标注推荐选项。如果你不确定技术细节，选推荐的不会错。

💬 用你习惯的语言沟通

你不需要说 "请使用 REST API 并实现 CRUD 操作"。说 "我需要能增删改查反馈数据" 就可以了。Claude 会翻译成技术方案。

⏲ 大功能拆小，分批做

如果你想做一个很大的功能（比如 "做一个完整的项目管理系统"），Claude 会建议你拆成几个子项目。建议听从它的建议 — 小步快跑比一口吃成胖子更靠谱。

📖 保存好设计文档

设计文档和实现计划是宝贵的资产。它们记录了 "为什么这样做" 的决策过程，对未来的维护和迭代非常有价值。

FAQ

产品经理常见问题

我完全不懂代码，能用 Superpowers 吗？▶

完全可以。Superpowers 的设计理念之一就是让 AI 像专业工程师一样工作，而你只需要扮演 "需求方" 的角色 — 告诉 Claude 你想要什么，回答它的澄清问题，审批设计方案。所有技术决策由 Claude 处理。

整个过程要多长时间？▶

取决于功能的复杂度。一个简单功能（比如一个反馈表单）：
头脑风暴：10-20 分钟（需要你参与）
计划编写：5-10 分钟（基本自动）
实现开发：30 分钟到数小时（完全自主）
收尾：2-5 分钟（需要你选择）

你实际需要参与的时间大约 20-30 分钟。实现阶段你可以去做其他事情。

我可以让 Claude 跳过某些步骤吗？▶

可以，你的指令优先级最高。你可以说 "不需要头脑风暴，直接做"。但不建议这样做 — 头脑风暴阶段正是确保 Claude 理解你需求的关键环节，跳过可能导致做出来的东西不是你想要的。

Claude 实现的时候我需要盯着吗？▶

不需要。设计和计划批准后，Claude 可以自主运行数小时。它有自己的质量保障流程（测试 + 双阶段审查），不需要你实时监督。你回来后可以查看结果，让 Claude 汇报完成情况。

"测试驱动开发" 是什么？为什么很重要？▶

简单地说：在写功能代码之前，先写 "检验功能是否正确" 的测试代码。

为什么重要？就像建房子前先画好图纸，确保每面墙、每根管道都在正确的位置。当代码有变动时，这些测试会自动运行，告诉你有没有不小心弄坏了什么。

对你意味着什么？Claude 交付的代码质量更高、更可靠。未来修改代码时，测试会自动检测出问题，避免 "改了这里坏了那里" 的情况。

"双阶段审查" 是什么？▶

想象一下工厂的两道质检：

第一道（规格合规）："检查做的东西对不对" — 一个独立的检查员拿着设计图纸，逐条对照实际产品，确认没有遗漏、没有多做、没有做错。

第二道（代码质量）："检查做工好不好" — 另一个检查员检查代码的组织、可读性、健壮性，就像检查焊接是否牢固、表面是否光滑。

两个检查员是独立的 — 第一个只管 "对不对"，第二个只管 "好不好"。这比一个人又当运动员又当裁判可靠得多。

"子代理" 是什么？▶

你可以理解为 Claude 的 "分身术"。

主 Claude 是项目经理，它会分派多个 "小 Claude"（子代理）去完成具体的编码任务。每个子代理是全新的、专注于一个任务的。做完后向主 Claude 汇报，主 Claude 再派审查子代理检查。

好处是：每个子代理都很专注，不会被其他任务的信息干扰，出错概率更低。

做出来的代码，之后可以让真人工程师维护吗？▶

完全可以。Superpowers 产出的代码比普通 AI 生成的代码更适合人类维护：

- 有完整的设计文档解释为什么这样做
- 有详细的实现计划说明每个部分的用途
- 有自动化测试保障修改安全
- 代码经过质量审查，符合工程标准

工程师接手时不需要猜 "这段代码想做什么" — 打开设计文档就能看到完整的决策过程。

会花多少钱？▶

Superpowers 插件本身是免费的（MIT 开源协议）。费用来自 Claude 的 API 调用。由于 Superpowers 会分派多个子代理，API 使用量会比不用 Superpowers 时更高。

典型参考：一个子代理完成一个小任务大约花费 $0.05 - $0.15。一个中等复杂度的功能可能有 5-10 个任务，加上审查，总计约 $1 - $3。

但考虑到质量提升（有测试、有审查、有文档），性价比很高。

如果 Claude 做的东西不是我想要的怎么办？▶

这正是 Superpowers 的价值所在 — 它通过分阶段确认大大降低了这种风险：

- 在头脑风暴阶段，你已经确认了需求
- 在方案选择时，你选了实现方式
- 在设计审批时，你确认了具体设计

如果到了收尾阶段才发现不满意，你可以选 "丢弃"，然后重新开始。由于代码在隔离的分支上开发，丢弃不会影响任何已有代码。

但如果你认真参与了前面的确认环节，很少会出现 "完全不是我想要的" 这种情况。
