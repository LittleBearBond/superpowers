在 Claude Code 中直接输入：

/plugin install superpowers@claude-plugins-official

这是最简单的安装方式，一条命令搞定。

先注册自定义市场，再安装：

How It Works

运行机制

理解 Superpowers 如何在每次会话中接管 Claude 的行为

第一步：会话启动注入

每次新会话启动时（包括 /clear 和 /compact），hooks.json 触发 session-start 脚本。该脚本读取 using-superpowers/SKILL.md 的完整内容，用 <EXTREMELY_IMPORTANT> 标签包裹后注入 Claude 的上下文。

// hooks.json 配置
{
  "hooks": {
    "SessionStart": [{
      "matcher": "startup|clear|compact",
      "hooks": [{
        "type": "command",
        "command": "\"${CLAUDE_PLUGIN_ROOT}/hooks/run-hook.cmd\" session-start",
        "async": false  // 同步执行，确保注入完成后才开始对话
      }]
    }]
  }
}

第二步：技能强制检查

注入的 using-superpowers 技能建立了核心规则：

在任何回复或操作之前，必须检查并调用相关技能。即使只有 1% 的适用概率也必须调用。这是非可选的、不可协商的。

Claude 在 Claude Code 中使用 Skill 工具调用技能。调用后技能内容被加载并展示给 Claude，Claude 随后按技能要求执行。

第三步：指令优先级

优先级来源说明
最高用户直接指令CLAUDE.md 配置、对话中的直接要求
中Superpowers 技能覆盖 Claude 的默认行为
最低默认系统提示Claude 的出厂行为

这意味着你可以通过直接指令覆盖任何技能行为（如 "不需要头脑风暴，直接做"），但 Superpowers 的价值正在于这些流程。

红旗检测表

如果 Claude 产生以下想法，using-superpowers 要求它立即停下来检查技能：

合理化想法正确做法
"这只是个简单问题"问题也是任务，检查技能
"我需要先了解更多上下文"技能检查在澄清提问之前
"先让我看看代码"技能告诉你如何探索
"这不需要正式的技能"如果技能存在，就用
"技能太小题大做了"简单的事情会变复杂
"我先做这一件事"做任何事之前先检查
"我记得这个技能的内容"技能会迭代更新，读当前版本

技能链式触发：brainstorming 完成后自动调用 writing-plans → writing-plans 完成后让用户选择 subagent-driven-development 或 executing-plans → 开发过程中自动调用 TDD、debugging、verification → 全部完成后自动调用 finishing-a-development-branch。你不需要手动触发每个技能。
Phase 1

头脑风暴 Brainstorming

整个工作流的起点。每当你要 "构建"、"创建"、"修改" 任何东西时自动激活

1

你会经历的 9 个步骤

superpowers:brainstorming ▶
硬性门禁：设计文档未经你的审批，Claude 禁止写任何代码、搭建任何脚手架、执行任何实现操作。无论项目多 "简单" 都必须经过此流程。
步骤 1 — 探索项目上下文

Claude 会先查看你的项目文件、文档、最近的 git 提交，了解项目现状。

步骤 2 — 提供可视化伴侣（可选）

如果你要做的事涉及 UI、图表、布局等视觉内容，Claude 会单独发一条消息询问是否启动浏览器端的可视化工具。这条消息不会与其他内容混合。

可视化伴侣是一个本地 Node.js 服务器，会在浏览器中展示 Mockup、线框图、方案对比。不涉及视觉的项目会跳过此步。
步骤 3 — 逐个提问澄清需求

Claude 会每次只问一个问题，偏好选择题格式。重点了解：

  • 你要做什么、为什么
  • 约束条件（技术栈、性能要求、兼容性）
  • 成功标准（怎样算 "做好了"）
如果项目范围过大（涉及多个独立子系统），Claude 会建议拆分为多个子项目，然后从第一个开始头脑风暴。
步骤 4 — 提出 2-3 种方案

Claude 始终提出 2-3 种实现方案，每种附带优劣分析和推荐理由。不会只给一个方案。

步骤 5 — 分段展示设计

设计不会一次性丢给你。Claude 会分段展示：架构、组件、数据流、错误处理、测试策略等。每段展示后会问 "这部分看起来对吗？"，你批准后才展示下一段。

段落大小根据复杂度自动调整：简单的几句话，复杂的 200-300 字。

步骤 6 — 编写设计文档

所有设计确认后，Claude 将完整设计保存到：

docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md

并自动 git commit。你可以在 CLAUDE.md 中自定义保存路径。

步骤 7 — 规格自审

Claude 派发一个子代理审查已写的设计文档，检查四项：

  • 占位符扫描 — 有无 TBD、TODO、不完整部分？
  • 内部一致性 — 是否有矛盾？
  • 范围检查 — 是否聚焦到一个计划能搞定？
  • 歧义检查 — 需求是否可能被两种方式解读？
步骤 8 — 你审阅设计文档

Claude 会说：

规格已写入并提交到 <path>。请审阅并告知是否需要修改。

等待你的回复。你可以要求修改（回到步骤 5 循环），也可以批准继续。

步骤 9 — 过渡到编写计划

你批准后，Claude 自动调用 superpowers:writing-plans 技能。这是 brainstorming 之后唯一的下一步 — 不会跳到实现。

Phase 2

创建工作树 Git Worktrees

在隔离的分支上工作，不影响主分支

2

隔离工作空间的创建流程

superpowers:using-git-worktrees ▶
目录选择（优先级顺序）
  1. 检查现有目录 — 项目中已有 .worktrees/（优先）或 worktrees/ 目录？
  2. 检查 CLAUDE.md — 项目是否配置了 worktree 目录偏好？
  3. 询问你 — 提供两个选择：
    .worktrees/（项目本地隐藏目录）
    ~/.config/superpowers/worktrees/<project>/（全局位置）
安全检查

如果选择了项目本地目录，Claude 会用 git check-ignore -q 验证目录是否被 gitignore。如果没有，先添加到 .gitignore 并提交。

创建步骤
  1. 检测项目名 — basename "$(git rev-parse --show-toplevel)"
  2. 创建 worktree — git worktree add "$path" -b "$BRANCH_NAME"
  3. 自动项目初始化 — 检测到 package.json → npm install，Cargo.toml → cargo build，requirements.txt → pip install 等
  4. 验证测试基线 — 运行测试套件，确保在干净状态下全部通过
  5. 报告位置 — 告知工作树路径和测试结果
如果基线测试失败，Claude 会报告失败详情并询问是否继续还是先调查问题。
Phase 3

编写实现计划 Writing Plans

将设计文档拆解为可由子代理独立执行的小任务

3

计划编写规范

superpowers:writing-plans ▶
编写原则

计划的假设是：执行者对代码库零上下文且品味存疑。所以每个任务必须包含一切所需信息。

任务粒度

每个任务控制在 2-5 分钟。以下每一项都是独立的步骤：

  • "编写失败测试" — 一个步骤
  • "运行测试确认失败" — 一个步骤
  • "编写最小通过代码" — 一个步骤
  • "运行测试确认通过" — 一个步骤
  • "提交" — 一个步骤
任务结构模板
### Task N: [组件名称]

**Files:**
- Create: `exact/path/to/file.py`
- Modify: `exact/path/to/existing.py:123-145`
- Test: `tests/exact/path/to/test.py`

- [ ] **Step 1: 编写失败测试**
```python
# 实际的测试代码，不是占位符
def test_feature_does_x():
    result = feature(input)
    assert result == expected
```

- [ ] **Step 2: 运行测试确认失败**
Run: `pytest tests/path/test.py::test_name -v`
Expected: FAIL with "function not defined"

- [ ] **Step 3: 编写最小实现**
```python
# 实际的实现代码
def feature(input):
    return expected
```

- [ ] **Step 4: 运行测试确认通过**
Run: `pytest tests/path/test.py::test_name -v`
Expected: PASS

- [ ] **Step 5: 提交**
```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
以下内容出现在计划中即为失败：TBD、TODO、"待定"、"添加适当的错误处理"、"编写上述代码的测试"（不含实际测试代码）、"类似 Task N"（必须重复代码）、没有代码块的代码步骤。
保存位置
docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md
计划自审（3 项检查）
  • 规格覆盖 — 逐条对照规格需求，指出哪个任务实现了哪条需求，列出遗漏
  • 占位符扫描 — 搜索所有红旗模式
  • 类型一致性 — 跨任务检查类型、方法签名、属性名是否匹配
执行方式选择

计划完成后，Claude 让你选择执行方式：

子代理驱动（推荐）

为每个任务分派独立子代理，任务间自动审查。快速迭代，适合大多数项目。

superpowers:subagent-driven-development

内联执行

在当前会话中逐步执行，分批处理，每批后等待你确认。

superpowers:executing-plans

实现者子代理

使用 Claude Code 的 Task 工具（即 Agent 工具）分派。Claude 将任务的完整文本粘贴到提示中（而不是让子代理自己读取计划文件）。子代理需要：

  • 按任务规格实现功能
  • 遵循 TDD 循环编写测试
  • 验证实现可用
  • 提交代码
  • 自审（完整性、质量、纪律、测试）
  • 报告状态：DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED
子代理被鼓励在不确定时提问而不是猜测。如果遇到架构决策、超出范围的重构、或无法理解的代码，它应该报告 BLOCKED 或 NEEDS_CONTEXT。
实现者状态处理
状态含义Claude 的处理
DONE任务完成直接进入规格合规审查
DONE_WITH_CONCERNS完成但有顾虑读取顾虑内容。正确性/范围问题 → 先处理再审查；观察性备注 → 记录后继续
NEEDS_CONTEXT需要更多上下文提供缺失的上下文，重新分派
BLOCKED被阻塞评估原因：上下文不足 → 补充上下文；推理能力不足 → 用更强模型；任务太大 → 拆分；计划有误 → 报告给你
双阶段审查（顺序不可颠倒）

第一阶段：规格合规审查

派发独立审查子代理，核心原则是不信任实现者的报告：

  • 不相信实现者说了什么
  • 不相信实现者声称的完整性
  • 不相信实现者对需求的解读
  • 必须读实际代码，逐行与需求对比

检查：是否遗漏需求？是否多做了不需要的？是否误解了需求？

第二阶段：代码质量审查

只有在规格合规通过后才进行。检查：

  • 代码质量（关注点分离、错误处理、类型安全、DRY）
  • 架构设计（SOLID、可扩展性、安全性）
  • 测试质量（测试逻辑而非 mock、边界情况、全部通过）
  • 文件职责（每个文件一个职责、接口清晰）

问题分级：Critical（必须修复）/ Important（应该修复）/ Minor（建议）

审查发现问题 → 实现者修复 → 审查者重新审查 → 循环直到通过。绝不跳过审查、绝不 "差不多就行"。
智能模型选择
任务类型选择模型典型场景
机械性任务廉价/快速模型改 1-2 个文件、规格明确的实现
集成任务标准模型跨多文件、需要判断力
架构/审查最强模型设计决策、代码审查

典型子代理任务成本：$0.05 - $0.15

绝对禁止的行为
  • 在 main/master 上直接实现（未经你同意）
  • 跳过任何一阶段的审查
  • 带着未修复的问题继续
  • 并行分派多个实现子代理（会冲突）
  • 让子代理自己读取计划文件（应粘贴完整文本）
  • 在规格合规审查之前启动代码质量审查

每一步的规则
步骤要求常见错误
RED一个最小测试，测试一个行为，清晰命名，用真实代码（不是 mock）测试名含 "and"（应拆分）、mock 过多

GREEN写能让测试通过的最简单代码，不添加额外功能、不重构、不 "改进"过度实现、"顺便" 加功能

REFACTOR只在绿色后重构，消除重复、改善命名、提取帮助函数。不添加行为在红色阶段重构、重构时加了新功能
合理化借口检测

以下想法出现时，Claude 被要求立即停下来回到 TDD：

  • "这太简单了不需要测试" → 如果简单，测试写起来也快
  • "我先写代码再补测试" → 先通过的测试证明不了什么
  • "这是重构不需要新测试" → 测试确认重构没破坏行为
  • "TDD 太教条了" → TDD 是务实的（发现 bug、防止回归、记录行为）
  • "手动测试过了" → 手动测试不可重复、没有记录
验证清单（8 项）

TDD 结束时必须能勾选所有项：

  • 每个函数都有测试
  • 每个测试在实现前都看它失败过
  • 每个测试的失败原因是预期的
  • 每次只写了通过测试的最小代码
  • 所有测试都通过
  • 测试输出干净
  • 测试用真实代码（不是 mock）
  • 边界情况已覆盖

无法全部勾选？说明跳过了 TDD。重来。

Phase 6

系统性调试 Debugging

遇到任何 Bug 或测试失败时自动激活 — 禁止盲目修复

6

四阶段根因分析法

superpowers:systematic-debugging ▶
铁律：不做根因分析就不修复。Phase 1 未完成，不允许提出修复方案。
Phase 1 — 根因调查
  1. 仔细阅读错误信息 — 完整阅读堆栈跟踪，不跳过
  2. 稳定复现 — 确定精确复现步骤，每次都能复现？
  3. 检查近期变更 — git diff、依赖更新、配置变更、环境差异
  4. 多组件系统中收集证据 — 在每个组件边界添加诊断，在提出修复之前
  5. 追踪数据流 — 错误的值从哪里来？持续向上追踪直到源头
Phase 2 — 模式分析
  1. 在代码库中找到正常工作的例子
  2. 与参考实现完整对比（不是浏览，是完整阅读）
  3. 列出每一个差异
  4. 理解依赖关系
Phase 3 — 假设检验
  1. 形成单一假设 — "我认为 X 是因为 Y"
  2. 最小化测试 — 最小的改动，一次只改一个变量
  3. 验证后才继续 — 有效 → Phase 4；无效 → 新假设（不堆叠修复）
  4. 不知道就说不知道
Phase 4 — 实施修复
  1. 创建失败测试（使用 TDD 技能）
  2. 实施单一修复 — 一次只改一个地方，不 "顺便" 改进
  3. 验证修复
  4. 修复无效且尝试 < 3 次 → 回到 Phase 1
  5. 3+ 次修复失败 → 必须停下来质疑架构是否正确
3+ 修复规则：如果已经尝试 3 次以上修复仍然失败，必须停下来讨论：模式本身是否正确？是不是因为惯性在坚持错误的方向？应该重构还是继续修补？
附带工具
  • root-cause-tracing.md — 沿调用栈回溯方法
  • defense-in-depth.md — 4 层验证（入口、业务逻辑、环境守卫、调试插桩）
  • condition-based-waiting.md — 用条件轮询替代固定超时
  • find-polluter.sh — 二分法定位哪个测试导致了污染

效果对比：系统性调试 15-30 分钟解决，首次修复成功率 95%。随机尝试 2-3 小时，成功率 40%。

Phase 7

完成前验证 Verification

在声称任何事情 "完成" 之前的最后一道关卡

7

验证五步法

superpowers:verification-before-completion ▶
铁律：没有新鲜验证证据就不声称完成。跳过任何一步 = 撒谎。
门禁函数（每次声称完成前运行）
  1. IDENTIFY — 什么命令能证明这个声明？
  2. RUN — 执行完整命令（新鲜的，不是缓存的）
  3. READ — 完整阅读输出，检查退出码，计算失败数
  4. VERIFY — 输出是否确认声明？否 → 说明实际状态并附证据；是 → 继续
  5. ONLY THEN — 做出声明
常见声明 vs 所需证据
声明需要的证据不够的
"测试全通过"测试命令输出：0 failures上次运行结果、"应该通过"
"Bug 已修复"复现步骤 → 无法复现"改了代码应该修好了"
"构建成功"构建命令输出：exit 0"没看到错误"
"需求已实现"逐条对照计划，每条有对应代码"大部分都做了"
"子代理完成了"检查 VCS diff、验证实际变更信任子代理的报告
红旗信号
  • 使用 "应该"、"大概"、"似乎" 等词
  • 验证前就表达满意（"太棒了！"、"完美！"、"搞定！"）
  • 准备 commit/push/PR 但还没运行验证
  • 信任代理/子代理的成功报告
Phase 8

代码审查 Code Review

请求审查和接收反馈的完整流程

8

审查请求 & 反馈处理

superpowers:requesting-code-review ▶
请求审查（3 步）
  1. 获取 Git SHA 
    BASE_SHA=$(git rev-parse HEAD~1)  # 或 origin/main
HEAD_SHA=$(git rev-parse HEAD)
  2. 分派审查子代理 — 使用 Task 工具，填充审查模板的 5 个占位符：
    {WHAT_WAS_IMPLEMENTED} 实现了什么
    {PLAN_OR_REQUIREMENTS} 计划/需求
    {BASE_SHA} 基础提交
    {HEAD_SHA} 当前提交
    {DESCRIPTION} 任务描述
  3. 处理反馈 — Critical 立即修复 / Important 继续前修复 / Minor 记录待办 / 审查员错了则反驳
审查子代理检查清单
  • 代码质量 — 关注点分离、错误处理、类型安全、DRY、边界情况
  • 架构 — SOLID、可扩展性、性能、安全
  • 测试 — 测试逻辑（非 mock）、边界、集成、全部通过
  • 需求对齐 — 所有计划需求已实现、无范围蔓延
  • 生产就绪 — 迁移策略、向后兼容、文档
接收反馈的规则

调用 superpowers:receiving-code-review 后：

  • 禁止表演性附和 — 不说 "说得太对了！"、"感谢指出！"、"Great point!"
  • 先验证再实施 — 确认建议确实是改进
  • 错了就反驳 — 用技术论据解释为什么不同意
  • YAGNI 检查 — 对 "更专业" 的建议评估是否真的需要
Phase 9

分支收尾 Finishing

所有任务完成且测试通过后的最后一步

9

四选一收尾操作

superpowers:finishing-a-development-branch ▶
流程
  1. 验证测试 — 运行测试套件。失败则停止，不能继续
  2. 确定基础分支 — git merge-base HEAD main
  3. 精确提供 4 个选项（不多不少）
  4. 执行你的选择
  5. 清理工作树
四个选项详解
选项操作工作树
1. 本地合并 切换到基础分支 → 拉取最新 → 合并功能分支 → 在合并结果上再次验证测试 → 删除功能分支 删除
2. 创建 PR git push -u → gh pr create（含摘要和测试计划） 删除
3. 保持原样 报告分支和工作树位置，不做任何操作 保留
4. 丢弃 需要你输入 "discard" 确认 → 显示分支名、所有提交、工作树路径 → 强制删除分支 删除

FAQ

常见问题

安装后 Claude 没有自动使用技能？▶

必须启动新会话。技能通过 session-start 钩子在新会话启动时注入，旧会话不会生效。用 /clear 也能触发重新注入。

能跳过头脑风暴直接写代码吗？▶

可以。用户指令优先级最高。直接说 "不需要头脑风暴，直接实现 X" 即可。但 Superpowers 的价值正在于这些流程 — 跳过会降低输出质量。

子代理开发和执行计划有什么区别？▶

子代理驱动开发：每个任务用新鲜子代理、自动双阶段审查、更快更自动化（推荐）。
执行计划：在当前会话逐步执行、分批处理、每批后等你确认。适合需要更多人工介入的场景。

子代理一个任务大概花多少钱？▶

典型成本 $0.05 - $0.15 每个任务。可以用内置分析脚本查看详细 Token 用量：

python3 tests/claude-code/analyze-token-usage.py ~/.claude/projects/<project-dir>/<session-id>.jsonl
Claude 是用什么工具调用技能的？▶

在 Claude Code 中使用 Skill 工具。Claude 调用后，技能内容会被加载并展示给它。它不会用 Read 工具读取技能文件 — 必须通过 Skill 工具。

"你的人类伙伴" 是什么意思？▶

Superpowers 刻意使用 "your human partner"（你的人类伙伴）而非 "the user"（用户）。这是精心设计的行为塑造语言 — 让 AI 代理将你视为合作伙伴而非服务对象，影响其决策方式和沟通风格。

设计文档和实现计划保存在哪？▶

默认位置：
设计文档：docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md
实现计划：docs/superpowers/plans/YYYY-MM-DD-<feature>.md
你可以在 CLAUDE.md 中自定义这些路径。

如何创建自己的技能？▶

在 ~/.claude/skills/ 下创建目录和 SKILL.md 文件。SKILL.md 需要 YAML 前置数据（name + description）。description 以 "Use when..." 开头，描述触发条件。使用 superpowers:writing-skills 技能来开发和测试新技能。

Windows 上钩子不工作？▶

Superpowers 使用多语言包装器 run-hook.cmd 支持 Windows。它会自动查找 Git Bash（C:\Program Files\Git\bin\bash.exe）。如果 Git 安装在非默认路径，需要手动更新。常见问题：
- "bash is not recognized" → Git Bash 路径不对
- 脚本在文本编辑器中打开 → hooks.json 指向了 .sh 文件而非 .cmd
